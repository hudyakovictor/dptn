# Гайд: система хронологического анализа лица на базе 3DDFA-V3

## 0. Ключевая идея, которую нужно понять до написания кода

3DDFA-V3 — это не «детектор лица», а **регрессор параметров 3DMM (Basel Face Model)**. Для каждого фото она возвращает *разложенный* набор коэффициентов:

| Блок | Размерность | Что описывает
|-----|-----|-----
| `id` (α_id) | 80 | **Геометрия черепа/лица конкретного человека**
| `exp` (α_exp) | 64 | Мимика (улыбка, прищур, открытый рот)
| `angle` | 3 | Поза головы: **yaw / pitch / roll**
| `trans` | 3 | Смещение головы в кадре
| `tex` (β) | 80 | PCA-текстура кожи
| `gamma` | 27 | Освещение (сферические гармоники)


Это и есть ответ на вашу проблему: **поза, мимика и освещение уже отделены от формы лица**. Вам не нужно «выравнивать наклон головы» на фотографиях — вы сравниваете не фотографии, а восстановленные 3D-формы, из которых поза математически исключена. Наклон головы попадает в `angle`, а не в `id`.

Итоговая стратегия:

1. `angle` → используем для **классификации на 9 ракурсов** (и как метрику качества кадра).
2. `id` (+ нейтрализованный 3D-меш) → используем для **сравнения геометрии лица по хронологии**.
3. Ракурсные корзины остаются нужны — но уже не для «честного сравнения», а для контроля систематической ошибки реконструкции (см. §6).

## 2. Пакетный прогон датасета (1800+ фото)

`demo.py` сохраняет визуализации и `.npy` с ландмарками, но для вашей задачи нужны **сырые коэффициенты**. Напишите свой батч-скрипт поверх внутреннего API (класс модели из `demo.py`), который для каждого фото сохраняет JSON/NPZ:

```python
# extract_all.py — идея скрипта (поверх кода 3DDFA-V3)
import os, json, numpy as np
# ... инициализация модели как в demo.py (face_model, recon_model, detector) ...

results = []
for fname in sorted(os.listdir(DATASET)):
    date = parse_date_from_filename(fname)   # у вас дата в имени файла
    img = load(fname)
    try:
        crop, trans_params = crop_face(img)          # RetinaFace + crop до 224x224
        coeffs = recon_model.predict(crop)           # словарь коэффициентов
    except NoFaceError:
        log_skip(fname); continue

    yaw, pitch, roll = np.degrees(coeffs['angle'].flatten())
    results.append({
        "file": fname,
        "date": date.isoformat(),
        "yaw": float(yaw), "pitch": float(pitch), "roll": float(roll),
        "id": coeffs['id'].flatten().tolist(),      # 80 чисел — главное!
        "exp": coeffs['exp'].flatten().tolist(),
        "face_size_px": crop_size(trans_params),    # для фильтра качества
    })

json.dump(results, open("coeffs_all.json", "w"))
```

Дополнительно сохраняйте **нейтрализованный меш**: восстановите вершины BFM с `exp = 0` и без поворота:

```python
# shape = mean_shape + id_basis @ alpha_id + exp_basis @ alpha_exp
# нейтральная форма (без мимики, без позы):
verts_neutral = mean_shape + id_basis @ alpha_id   # (35709, 3)
np.save(f"meshes/{fname}.npy", verts_neutral.astype(np.float32))
```

Это лицо человека «в анфас, без выражения, в канонической системе координат» — независимо от того, как он был снят.

## 3. Классификация на 9 ракурсов — теперь автоматическая

Забудьте про ручную сортировку: yaw из `angle` даёт точную корзину. Пример разбиения (пороги подгоните под распределение вашего датасета гистограммой):

```python
def bucket(yaw):
    a = abs(yaw); side = "L" if yaw < 0 else "R"
    if a < 10:  return "frontal"
    if a < 22:  return f"34_light_{side}"
    if a < 40:  return f"34_middle_{side}"
    if a < 65:  return f"34_deep_{side}"
    return f"profile_{side}"
```

Бонусы:

- `pitch` и `roll` тоже известны → можно **отбраковывать** кадры с сильным задиранием/опусканием головы (|pitch| > 20°), где реконструкция подбородка/лба хуже.
- Постройте гистограмму yaw по всему датасету — увидите, честно ли распределены ваши 9 корзин, и где пороги стоит сдвинуть.

## 4. Фильтрация качества (критично для форензики)

Мусор на входе = сенсация на пустом месте. Отбрасывайте или помечайте флагом:

- лицо в кропе меньше ~120–150 px по межзрачковому расстоянию;
- сильный блюр (variance of Laplacian по кропу ниже порога);
- |pitch| > 20°, |roll| > 25°;
- окклюзии — тут поможет `--seg_visible`: 3DDFA-V3 умеет отдавать сегментацию 8 частей лица с маской видимости. Если видимость глаз/носа/рта < порога (рука, микрофон, очки-маска) — кадр в карантин;
- экстремальная мимика: `‖exp‖` в верхних 5% — нейтрализация мимики на таких кадрах менее надёжна.

## 5. Сравнение по хронологии: три уровня доказательности

### Уровень A — вектор идентичности (быстрый скрининг)

Косинусное расстояние между 80-мерными `id`-векторами:

```python
def dist(a, b):
    a, b = np.asarray(a), np.asarray(b)
    return 1 - a @ b / (np.linalg.norm(a) * np.linalg.norm(b))
```

Постройте:

- **временной ряд**: расстояние каждого фото до центроида всех `id` (или скользящего центроида за ±1 год) по датам;
- **матрицу попарных расстояний** 1800×1800 с сортировкой по дате — «блоки» на тепловой карте видны глазом;
- **кластеризацию** (HDBSCAN / agglomerative) на `id`-векторах. Если «двойники» существуют как устойчивые кластеры — они проявятся; затем проверяете, коррелируют ли кластеры с датами/событиями.


### Уровень B — геометрия меша (тяжёлая артиллерия)

Для пар/кластеров, подозрительных на уровне A: берёте нейтрализованные меши (35 709 вершин), выравниваете **Procrustes-ом** (scipy `procrustes` или Umeyama с масштабом) и считаете per-vertex ошибку. Ключевой приём — считать её **по зонам**, используя part-маски из 3DDFA-V3 (глаза, нос, губы, брови и т.д.):

```python
for part, idx in part_masks.items():        # маски вершин из assets
    err = np.linalg.norm(A[idx] - B[idx], axis=1).mean()
```

Так вы получаете форензически интерпретируемое утверждение: «форма носа стабильна, а геометрия скуловой зоны в период X отклоняется на N сигм».

### Уровень C — антропометрические инварианты

Из 68/106/134 3D-ландмарок (не 2D! именно 3D-координаты канонического меша) считайте **отношения** расстояний, которые не меняются с возрастом и весом почти никак: межзрачковое / ширина носа у основания, высота лица / бизигоматическая ширина и т.п. Отношения нечувствительны к масштабу и суммируются в компактный «паспорт лица» на каждую дату.

## 6. Главный методологический капкан: без калибровки выводы ничтожны

Всё вышеописанное даст числа. Числа без базовой линии — это не расследование, а нумерология. Обязательные контроли:

1. **Within-person baseline.** Возьмите 3–5 контрольных персон (другие публичные фигуры с большими фотоархивами за те же годы), прогоните через тот же пайплайн. Так вы узнаете *естественный разброс* `id`-векторов одного человека из-за возраста, веса, качества съёмки. Аномалия у целевой персоны считается аномалией, только если выходит за этот коридор.
2. **Ракурсный биас.** Реконструкция с deep-3/4 и профиля систематически хуже фронтальной — `id`-вектора с одного и того же лица, снятого с разных ракурсов, будут дрейфовать. Поэтому: (а) сравнивайте распределения **внутри одной корзины ракурса** между периодами, (б) в смешанном анализе добавляйте yaw как ковариату или регрессируйте его влияние.
3. **Возраст ≠ двойник.** За 20+ лет мягкие ткани меняются сильно, костная основа — почти нет. Взвешивайте зоны: нос (спинка), орбиты, лоб — консервативны; щёки, подбородочная линия, веки — нет.
4. **Фокусное расстояние.** Слабослышимый убийца точности: телевик и селфи-ширик по-разному искажают перспективу лица, а 3DMM-фит частично «съедает» это в форму. Хотя бы группируйте по типу съёмки (пресс-пул/телефото vs близкая камера), если это восстановимо.


## 7. Независимая перекрёстная проверка

99-й уровень — это никогда не доверять одной модели. Параллельно прогоните датасет через 2D-эмбеддинги распознавания лиц (**ArcFace / insightface**, 512-мерный вектор): они устойчивы к позе по-своему (обучены на миллионах лиц) и дают независимый сигнал. Схема доказательства: *если и кластеры ArcFace, и кластеры 3DDFA-`id`, и зонная геометрия меша независимо показывают одну и ту же временную структуру — у вас есть результат. Если показывают разное — у вас есть артефакт пайплайна.*

## 8. Итоговая архитектура

```plaintext
photos/ (1800+, дата в имени)
   │
   ├─ 1. RetinaFace crop + фильтры качества (blur, размер, окклюзии по seg_visible)
   ├─ 2. 3DDFA-V3 → {id(80), exp(64), angle(3), ...} + нейтральный меш (exp=0, поза=0)
   ├─ 3. Бакетизация по yaw → 9 ракурсов (+ отбраковка по pitch/roll)
   ├─ 4. Уровень A: косинус по id → таймлайн, heatmap, HDBSCAN
   ├─ 5. Уровень B: Procrustes мешей → per-vertex / per-part ошибки
   ├─ 6. Уровень C: 3D-антропометрические отношения
   ├─ 7. Контроль: baseline-персоны + ArcFace кросс-проверка
   └─ 8. Отчёт: интерактивный таймлайн + тепловые карты + примеры пар
```

---

Резюмируя главное: не пытайтесь «докрутить головы» на фото — используйте то, что 3DDFA-V3 сама раскладывает лицо на форму/мимику/позу, сравнивайте нейтрализованные 3D-формы и `id`-вектора, а 9 ракурсов оставьте как средство контроля систематической ошибки, причём классифицируйте их автоматически по yaw. И ни один вывод не публикуйте без baseline-персон и независимой проверки вторым методом (ArcFace). Если хотите — могу следующим шагом написать готовый батч-скрипт `extract_all.py` под структуру репозитория 3DDFA-V3 и ноутбук для анализа таймлайна.

## КРИТИЧНО: пайплайн вообще не использует 3DDFA-V3

**1. `s1_extraction/engine.py` строит фейковую реконструкцию, а не 3D-лицо.**
У вас есть настоящий, качественно написанный `modules/reconstruction.py` (`ReconstructionAdapter` с 3DDFA_v3, кэшем, MD5, обнулением `exp`), а также `pose_estimator.py`, `alignment.py`, `quality_gate.py`, `visibility.py` — но **движок S1 не импортирует ни один из них**. Вместо этого он вызывает:

- `detect_face_bbox` — Haar-каскад `frontalface_default`, который **не находит профили и глубокие 3/4** (а это большинство вашего датасета). При провале — `fallback_face_bbox`: тупой центральный кроп 72% кадра.
- `build_placeholder_reconstruction` из `shared/utils.py` — синтетическая сетка-параболоид 12×14 вершин, где «лэндмарки» — это **фиксированные доли bbox**: `left_eye = x + 0.33*w`, `nose = x + 0.50*w` и т.д. Это не лицо, это прямоугольник.


**2. Поза берётся из имени файла, а не из фото.**
`pose_hint_from_name` ищет токены `left_threequarter_deep` и т.п. в имени файла. У вас в именах — **даты**. Значит на реальном датасете каждое фото получит `yaw=0, pitch=0, roll=0, bucket=UNKNOWN`. Проблема «разных наклонов головы», ради которой всё затевалось, не решается вообще: `pose_gap_deg` в S4 всегда будет 0, разбиение на 9 ракурсов не произойдёт.

**3. `reconstruction.py` молча деградирует в заглушку.**

```python
try:
    from model.recon import face_model
    _HAS_3DDFA = True
except ImportError:
    _HAS_3DDFA = False   # ← ни warning, ни ошибки
```

Плюс захардкоженный путь `CORE_3DDFA_ROOT = Path("/Users/victorkhudyakov/dutin/core/3ddfa_v3")` — на любой другой машине (или после переноса папки) импорт тихо падает, и пайплайн «успешно» отрабатывает на мусоре. Для форензики это худший из возможных режимов отказа: **должен быть hard fail**, а не тихий фолбэк.

**4. `aliases.py` фабрикует ~200 «анатомических» метрик из ничего.**
`project_geometry_aliases` берёт bbox-прокси (ширина лица = расстояние между двумя *константными* долями bbox) и арифметически смешивает их в метрики с названиями `zone_chin_normal_mean_x`, `brow_ridge_L_convexity_index`, `orbit_L_orbital_bowl_volume_proxy`... Например:

```python
"brow_ridge_L_convexity_index": clip01(0.28 + sym * 0.18 + ratio(nose_bridge, face_height, 0.14)),
```

Никакой надбровной дуги здесь не измеряется — это функция от размеров bbox. Дальше S3–S6 будут выносить «вердикты» про форму подбородка и орбит, **которые никогда не измерялись**. В отчёте это выглядит как анатомический анализ, но им не является — это самая опасная ошибка с точки зрения журналистской добросовестности.

**5. Whitelist уничтожает реальные метрики и оставляет только фейковые.**
`_filter_geometry_metrics` фильтрует по `GEOMETRY_CORE_METRICS` (там только `zone_*`, `chin_*`, `brow_ridge_*`...). Базовые измеренные значения (`face_width_px`, `mesh_depth_span`...) **выбрасываются**, выживают только сфабрикованные алиасы. При этом в S4 `_normalized_distance` есть ветка `key_weight = 1.15` для ключей `mesh_*/face_*` — **мёртвый код**, такие ключи туда никогда не попадают.

## СЕРЬЁЗНО

**6. Двойное вычитание возрастной поправки в S4.** `_normalized_distance` уже вычитает `age_shift` из дистанции, затем `_compare_pair` вычитает `age_explained_distance` из той же `geometry_distance` ещё раз. Систематически занижаете различия — прямой путь к ложному «same».

**7. Бессмысленные пороги в S3.** `_build_thresholds`: `texture_suspicion = mean(mean+std)` по сырым метрикам, где `texture_gray_mean` ≈ 128 → после `clip(…, 0.3, 0.9)` порог **всегда 0.9**. Смешаны величины разных размерностей с безразмерным порогом 0–1.

**8. Захардкоженные вердикты-персоны.** `identity_hint in {"PUT", "UDMURT", "VAS", "OTHER"}` в `s3_identity/engine.py` — выводы («существуют конкретные три двойника») зашиты в код *до* анализа. Это confirmation bias на уровне архитектуры: система может только подтвердить гипотезу, но не опровергнуть. Метки кластеров должны быть анонимными (`cluster_0`, `cluster_1`) и получаться *из данных*.

**9. Магические веса без калибровки.** В S4: `geometry_distance * 0.62 + texture_distance * 0.28 + ... / 3.6`, `clip(1.15 - q, 0.55, 1.35)` и десятки подобных. Ни один из них не выведен из calibration-датасета — а именно для этого S3 и существует.

**10. `expression_flags` — нонсенс.** `smile_excluded = mouth_width > 40.0` px, но лэндмарки рта — константные доли bbox, значит `mouth_width = 0.24*w` всегда. Флаг фактически означает «bbox шире 167 px», а не «улыбка».

**11. `stable_photo_id = Path.stem` + `rglob`.** Два файла с одинаковым именем в разных подпапках получат один `photo_id` и **молча перезапишут** артефакты друг друга в `output_dir`.

**12. Шум в S3 конфликтует с хронологией.** `_build_pairwise_noise` считает шум по соседним *по дате* парам — на калибровке за 20 лет в «шум» попадёт реальное старение, что потом раздует `noise_discount` и замаскирует настоящие различия.

## МЕЛКОЕ

- `config/pipeline.yaml` секция `paths:` **игнорируется** — `run.py` читает из конфига только ключи `s1`…`s6`; пути берутся из CLI/дефолтов `/Volumes/SDCARD/...`.
- В `aliases.py` дублируются ключи словаря (`zone_chin_normal_mean_z`, `zone_chin_span_depth_ratio`, `zone_nose_bridge_tip_bbox_area_ratio` определены дважды) — Python молча берёт последнее.
- `MetricsEngine._texture_metrics` (~70 строк) — мёртвый код, заменён `TextureExtractor`.
- `self.texture_extractor._quality_sensitive_excluded` — чтение приватного мутируемого состояния, разделяемого между фото; при исключении в `extract()` флаг «протечёт» от предыдущего фото.
- `datetime.utcnow()` — deprecated (Python 3.12+), используйте `datetime.now(timezone.utc)`.
- `metric_notes["geometry_space"] = "placeholder_canon"` — хардкод честно признаётся, что пространство фейковое, даже когда/если подключите реальный 3DDFA.
- `is_over_smoothed` (sharpness > 0.88 и низкий шум) помечает подозрительными самые качественные фото.

## Диагноз: почему ранние фото = «силикон»

Посмотрите на ваш собственный `TOP20_FEATURES`. Практически все топ-признаки — это **меры высокочастотной микротекстуры**, а не биологии кожи:

| Признак | Что реально измеряет | Что происходит на фото 1999 г.
|-----|-----|-----
| `glcm_dissimilarity_d2/d3/d5` (8 из 20!) | контраст между соседними пикселями | ↓ падает при низком разрешении/мягкости → «гладко»
| `glcm_homogeneity_d3/d5` | однородность соседних пикселей | ↑ растёт на размытых фото → «гладко»
| `homo_local_var_w15/w31_cv` | локальная дисперсия | ↓ на софт-фото
| `morph_tophat_r4/r8_std` | мелкие светлые детали (поры!) | ↓ поры не разрешаются камерой 2000-х
| `grad_sobel_mag_skewness` | распределение градиентов (края) | ↓ мало резких краёв
| `lbp_uniform_r5_std` | локальные бинарные паттерны | ↓ на низкочастотном изображении


Силикон в обучающей выборке выглядел «гладким» именно потому, что у него **нет пор и микрорельефа → низкая dissimilarity, высокая homogeneity**. Ровно ту же сигнатуру даёт **мыльная, низкодетальная, пережатая JPEG плёночная фотография 1999–2005 гг.**, даже если кожа на ней прекрасно видна глазом. Модель выучила не «кожа vs силикон», а **«много высокочастотных деталей vs мало»**, то есть по факту — **резкость и разрешение снимка**.

Это учебниковый случай **confounder / dataset shift**: обучались вы (почти наверняка) на современных высокодетальных фото, а применяете к архивным. `LOO-CV 96.70%` в `docstring` — это точность *на том распределении качества*, к вашему датасету она отношения не имеет.

## Решения — по возрастанию правильности

**1. Немедленный костыль — quality gating (обязателен в любом случае).**
Классификатор должен **отказываться выносить вердикт**, когда фото вне того диапазона качества, на котором он валиден. У вас в `classify()` уже проброшен `quality`, но он не используется. Введите порог по резкости/разрешению и возвращайте `unknown` вместо `silicone`:

```python
if quality["sharpness"] < SHARP_MIN or quality["face_px"] < PX_MIN:
    return {"texture_skin_hint": "unknown", "reason": "below_texture_valid_range", ...}
```

Это не решает задачу, но останавливает поток ложных «силиконов». Честный «не могу измерить» бесконечно ценнее ложного вердикта.

**2. Нормализация домена — привести все фото к общему «разрешению текстуры».**
Причина сбоя — разная эффективная детализация. Уравняйте её:

- считайте GLCM/LBP/top-hat **на фиксированном физическом масштабе** — ресемплируйте ROI щеки/лба так, чтобы межзрачковое расстояние всегда было, скажем, 300 px (у вас есть 3DDFA-лэндмарки — масштаб известен точно);
- перед метриками приводите изображения к общему частотному диапазону (low-pass до полосы худшей камеры), чтобы новые фото не «выигрывали» за счёт лишних деталей;
- матчьте JPEG: слегка пережимайте всё до общего уровня, чтобы артефакты сжатия были одинаковы.
После этого силикон должен оставаться гладким, а кожа — текстурной, независимо от эпохи.


**3. Регрессия конфаундера (decorrelation).**
Обучите вспомогательную модель `sharpness → каждая текстурная фича` и работайте с **остатками**. Тогда из dissimilarity удаляется всё, что объяснимо резкостью, и остаётся только «текстурная» компонента. Ваш `residual_bio_iqr` намекает, что вы к этому подбирались — доведите до конца для всех признаков.

**4. Переобучение на сбалансированной по качеству выборке (правильное решение).**
Пересоберите обучающий набор так, чтобы `real` и `silicone` были **сматчены по резкости, разрешению, году и сжатию** (matched sampling / добавьте деградированные версии реальных фото — downscale+JPEG для аугментации). Тогда модель физически не сможет использовать качество как признак. Проверяйте не общий CV, а **точность в срезе по годам** — 1999-2005 отдельно.

**5. Стратегически — усомниться в самой задаче.**
Отделить силикон от кожи по 2D-текстуре на **неконтролируемых архивных фото разных камер за 25 лет** — на грани возможного, и легко даёт псевдонаучные выводы. Микротекстура кожи (поры ~0.1 мм) требует разрешения, которого в архиве 1999 г. просто **нет физически** — информация о порах не «размыта», её там не записано. Поэтому:

- рассматривайте текстуру как **слабый вспомогательный сигнал**, а не доказательство;
- никогда не выносите «силикон» на фото ниже порога качества — только `unknown`;
- основную доказательную нагрузку держите на **геометрии/идентичности** (стабильные костные пропорции), где 3DDFA даёт устойчивый к качеству сигнал.
